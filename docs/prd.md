## 1. 产品概述

### 1.1 产品定位

一句话定位：**本系统不是“搜政策”的百度，而是 AI 产业的“政策芯片”**。  

系统聚焦人工智能相关政策（含 OPC 政策），面向 AI 产业链参与方（科技人才、科技企业、转型企业、园区运营方），通过画像标签识别、政策匹配计算、卡点诊断、成长导航与材料智能生成，形成“可查、可算、可办、可协同”的政策服务闭环。

### 1.2 覆盖政策范围

- **AI 专项政策**：人工智能+制造、AI 场景揭榜挂帅、大模型专项等。
- **AI 关联政策**：智改数转（含 AI 应用）、工业互联网（含 AI 技术）、数据要素（AI 数据合规）等。
- **AI 前置政策**：算法备案、数据安全评估、高新认定、科小入库等。
- **AI 人才政策**：AI 人才引进、人才补贴、创业扶持等。
- **OPC 政策**：面向 One Person Company（超级个体/一人公司）的创业启动资金、算力券/模型券/语料券、社区入驻优惠、场景开放、人才安居与培训等。

重点地区含：深圳、苏州、北京中关村、上海徐汇等 OPC 先行区域。

### 1.3 核心问题与解决方案

| 痛点 | 现状 | 解决方案 |
|---|---|---|
| AI 政策信息获取难 | 多部门分散发布，更新快，企业难跟踪 | 多源采集政策库 + AI 自动解读 |
| 政策匹配判断难 | AI 合规前置项与传统条件叠加，人工难梳理 | 画像比对 + 匹配度计算 + 逻辑锁检查 |
| 申报执行难 | 材料专业门槛高，重复填报严重 | 素材复用 + 材料自动生成 + 预审 |
| 成长规划难 | 不清楚“先做什么后做什么” | 成长导航仪 + 依赖图谱 |
| 产业协同机会少 | 联合申报缺渠道和信任机制 | 隐性标签池 + 协同撮合 + 联合申报识别 |
| OPC 创业选址难 | 各地政策分散、差异大 | OPC 政策统一归集 + 跨地区智能比较 |

### 1.4 目标用户

| 用户类型 | 典型角色 | 核心诉求 |
|---|---|---|
| 科技人才 | AI 工程师/研究员/创业者（含 OPC） | 人才补贴、安居、创业扶持、OPC 政策 |
| 科技企业 | AI 公司、软件技术服务商（含 OPC 企业） | 补贴申报、资质认定、客户协同、券类补贴 |
| 转型企业 | 有数字化改造需求的传统企业 | 智改数转补贴、找 AI 供应商 |
| 园区运营方 | AI 产业园/OPC 社区团队 | 精准招商、政策发布推送、产业洞察 |

> OPC 说明：OPC 不单列为独立系统角色，通过人才或企业画像中的 OPC 属性识别并触发专属策略。

### 1.5 核心价值

#### 面向企业与人才
- **算得清**：自动算可申报政策与预估金额。
- **看得懂**：解释可申报原因与卡点。
- **办得快**：材料一键生成，减少重复填报。
- **有方向**：成长路径建议可执行。
- **找得到**：发现联合申报与协同合作机会。

#### 面向园区
- **看得见**：产业结构与链路可视化。
- **找得准**：智能识别招商目标（含 OPC）。
- **引得来**：园区政策发布后触达全平台并形成入驻转化闭环。

## 2. 产品目标与范围

### 2.1 产品目标（MVP）

1. 建立可持续更新的 AI+OPC 政策库（多源采集）。
2. 完成画像驱动的政策匹配、卡点分析与成长导航。
3. 完成材料工厂（清单生成、自动填充、AI 预审、在线编辑循环）。
4. 打通园区端政策发布、推送、招商与产业洞察能力。
5. 建立协同申报推荐能力（企业/人才/园区多角色）。

### 2.2 范围边界（明确不做）

- 不替代政府官方申报系统，不直接提交到政务平台。
- 不与园区既有管理系统深度打通（入驻数据由园区自行维护）。
- 申报进度默认基于政策时间节点推算，非官方实时状态。

## 3. 总体架构与技术选型

### 3.1 技术选型（国产、开源、低成本）

| 类别 | 选型 | 说明 |
|---|---|---|
| 前端 | Vite + React 18 + TypeScript + Ant Design 5 | SPA 架构，Zustand 状态管理，Axios HTTP 客户端 |
| 后端 | FastAPI + SQLAlchemy 2.0 + Alembic | Python 异步框架，asyncpg 驱动，Pydantic v2 校验 |
| 数据库 | PostgreSQL 16（阿里云 RDS） | JSONB 存储画像子结构，支持全文检索 |
| 缓存 | 后续按需引入 Redis | 第一阶段利用 PostgreSQL + 应用层缓存 |
| 向量库 | 后续按需引入 Milvus | 第一阶段使用 pgvector 扩展或 API 侧检索 |
| 大模型 | 通义千问（Qwen，主力） / DeepSeek（备选） | 通过 OpenAI 兼容接口统一调用，可热切换 |
| OCR | 百度 OCR | 按量付费，证照识别 |
| 文件存储 | 本地文件系统（第一阶段） | 后续可迁移至 MinIO / OSS |
| 部署 | Docker + Docker Compose + Caddy | Caddy 自动 HTTPS + 反向代理 |

### 3.2 分层架构

- **用户层**：统一 Web（科技人才/科技企业/转型企业，按角色权限动态渲染）+ 园区 Web（独立布局）。
- **接入层**：Caddy（自动 HTTPS、反向代理、静态资源服务）。
- **智能体层**：交互导航、政策计算、材料工厂、产业协同、产业洞察（园区专用），共 5 个 AI Agent。
- **业务服务层**：FastAPI 提供 RESTful API，包含用户/政策/画像/申报/素材/园区等服务模块。
- **数据层**：PostgreSQL（业务数据 + JSONB 画像）+ 本地文件存储（素材文件）。
- **外部服务**：通义千问 API、百度 OCR、企查查 API 等第三方服务。

## 5.1 功能总览

### 5.2 用户中心

#### 5.2.1 注册登录
- 手机号+验证码注册
- 手机号+密码登录
- 验证码登录
- 忘记密码（短信重置）

#### 5.2.2 注册流程
1. 输入手机号并验证；
2. 选择角色；
3. 完善基础信息；
4. 完成注册并引导画像建设。

#### 5.2.3 角色引导策略
- 科技人才：引导完善人才画像，并询问是否为 OPC 创业者。
- 科技企业：引导完善企业画像，并询问是否为 OPC 企业。
- 转型企业：引导完善企业画像。
- 园区运营方：引导完善园区画像，并询问是否为 OPC 社区。

### 5.3 画像中心

#### 5.3.1 人才画像
- 基础信息、教育背景、职业信息、资质认定、成果产出、OPC 信息。
- OPC 关键字段：是否 OPC、创业阶段、入驻社区、AI 方向、算力需求。
- 规则：OPC 创业者场景下，创业阶段与 AI 方向为必填。

#### 5.3.2 企业画像
- 采集方式：手填、证照 OCR、数据授权、素材解析。
- 字段覆盖：基础信息、经营数据、资质、知识产权、AI 合规、通用合规、OPC 信息。
- **自动同步规则**：资质认定上传/填写后，同步更新合规状态判断。
- **完整度公式**：已填字段数 / 总字段数 × 100%。  
  - 优秀：≥80%（精准匹配）
  - 良好：60%-79%（基础匹配）
  - 待完善：<60%（结果偏差风险）

#### 5.3.3 园区画像
- 基础信息、产业定位、入驻情况、招商需求、OPC 社区信息。
- 说明：平台不承担园区企业管理，数据由园区自行维护，仅用于分析与推荐。

### 5.4 政策中心

#### 5.4.1 政策库

**入库机制**：多源自动采集 + AI 解析 + 可选人工审核。

数据源：
- 第三方政策数据服务（主源，通常 T+1）
- 定向爬虫（补充，每日）
- 园区上传（实时生效）
- 用户上报（实时补充）

入库流程：
1. 多源采集；
2. AI 筛选（保留 AI/OPC 相关）；
3. AI 去重（文号/标题/机构）；
4. AI 结构化解析（条件、支持、材料、流程）；
5. 入库生效（默认）；
6. 可选人工复核（高影响政策建议复核）。

筛选维度：
- 政策层级、支持类型、支持领域、适用对象、部门归口、状态、来源、OPC 相关、地区。

状态定义：
- 申报中 / 即将开始（≤30 天）/ 已截止 / 已失效。

#### 5.4.2 政策详情

详情区块：
- 基本信息
- AI 解读
- 我的匹配（登录后）
- 政策知识图谱（纵向派生 + 横向成长依赖）
- 申报条件（必要/加分/排除）
- 支持内容
- 材料与流程

### 5.5 智能匹配

#### 5.5.1 匹配计算

系统根据画像计算可申报政策并排序展示。

匹配度公式：
`匹配度 = Σ(条件权重 × 满足度) / Σ(条件权重) × 100%`

满足度规则：
- 完全满足 = 100%
- 部分满足 = 按比例
- 不满足 = 0%

状态判定：
- 完全匹配：100%
- 高度匹配：80-99%
- 部分匹配：50-79%
- 低匹配：<50%

预估金额：
`预估金额 = 基础金额 + 加分项金额`

#### 5.5.2 逻辑锁（前置项）一票否决机制

- 若任一必要条件（含 AI 合规/逻辑锁）未 100% 满足：  
  - 总匹配度最高封顶 49%，状态=有卡点/不可申报。
- 若逻辑锁全部通过：  
  - 基础分 70；  
  - 加分项 20；  
  - 材料完备度 10；  
  - 总分 = 70 + 20×加分达成比例 + 10×材料完备比例。

#### 5.5.3 OPC 特殊匹配规则

1. OPC 专属政策仅匹配 `is_opc=true` 用户。  
2. 政策要求入驻社区时，执行入驻前置检查：  
   - 已入驻：正常计算；  
   - 未入驻：可继续计算并标记“需先入驻 OPC 社区”，推荐 Top3 社区（基于 LBS + AI 方向）。  
3. 收益拆分展示算力券/模型券/语料券等类型。  
4. 支持按地区分组比较跨城政策性价比。

#### 5.5.4 卡点分析

输出维度：
- 卡点条件
- 当前值 vs 目标值
- 差距描述
- 改进建议
- 预计成本与周期

卡点类型：
- 数据卡点、资质卡点、合规卡点、材料卡点、时间卡点。

#### 5.5.5 成长导航

路径规划逻辑：
1. 检查 AI 合规 + 通用合规状态；
2. 扫描全量政策匹配；
3. 构建依赖图谱；
4. 识别“跳一跳够得着”（70-99%）；
5. 评估完成前置项后可解锁政策与收益；
6. 按投入产出比排序形成路径；
7. 输出步骤收益、投入、周期、窗口期。

典型依赖：
- 科小入库 → 高新认定 → 专精特新 → 小巨人
- 算法备案 → AI 专项政策
- 数据安全评估 → 数据要素政策
- 入驻 OPC 社区 → 社区专属政策

#### 5.5.6 产业协同与联合申报

协同类型：
- 单线主导型（甲方拿补贴，乙方技术配套）
- 多线共振型（同项目双方分别申报不同政策）
- 多方联合体（3-4 方及以上联合）

协同引擎基于隐性标签池（BP/审计报告/历史材料/对话）识别供需并推荐合作方。

### 5.6 申报中心

#### 5.6.1 素材管理

素材分类：证照、财务、资质、知识产权、人员、项目。  
核心能力：一次上传，多政策复用；OCR 提取；有效期管理；完整度提示。

关键数据字段（摘要）：
- `material_id`、`entity_id`、`category`、`file_path`
- `ocr_content`、`extracted_data`
- `valid_from`、`valid_to`、`status`

#### 5.6.2 材料生成（材料工厂）

流程：
1. 选择政策；
2. 读取模板库（政策入库时预解析）；
3. 匹配素材；
4. 检查必填与逻辑；
5. AI 文案生成（RAG + 追问补全）；
6. 自动填充；
7. 生成材料；
8. AI 深度预审；
9. 在线修改与再预审循环；
10. 可选人工专家预审；
11. 定稿并跳转官方申报平台。

AI 预审检查：
- 客观一致性
- 技术方案合理性
- 政策导向贴合度
- 历史驳回风险点

人工专家预审：可选增值，1-3 个工作日。

#### 5.6.3 申报记录

平台定位：记录“材料准备到结果反馈”全过程，不接入官方申报系统。

状态流转：
- 材料生成 → 预审完成 → 待前往申报 → 已前往申报 → 预估审核中 →（用户反馈）已通过/未通过/已退回 → 已拨付或修改重提

机制说明：
- 预估进度来自政策时间节点推算；
- 实际结果需用户上传官方截图反馈；
- 反馈可获积分奖励。

### 5.7 园区工作台

#### 5.7.1 智能招商
- 输入：园区画像（主导产业、入驻结构、补链需求）。
- 维度：产业方向匹配、补链价值、发展阶段、搬迁可能性、政策吸引力。
- 输出：招商目标列表 + 招商策略建议。

#### 5.7.2 园区政策发布与推送

流程：
1. 园区上传政策（PDF/Word/手填）；
2. AI 自动解析；
3. 园区确认修正；
4. 入库生效（全平台可见）；
5. 系统匹配全平台用户；
6. 园区选择推送对象；
7. 多渠道推送（站内/短信/邮件）。

关键业务规则：
- 园区政策上传即生效；
- 园区对内容准确性负责；
- 对未入驻用户自动附带园区介绍与“申请入驻”入口，形成招商闭环。

#### 5.7.3 产业洞察
- 生成产业图谱、企业分布、趋势分析报告。
- 数据底座：园区白名单（主）+ LBS 与工商信息泛抓取（辅）。

### 5.8 消息中心

消息类型：
- 政策提醒
- 匹配通知
- 申报进度
- 协同邀请
- 系统通知

触发方式：系统自动触发 + 用户行为触发 + 管理端公告。

## 6. 五大智能体设计

### 6.1 能力矩阵

| Agent | 核心职责 |
|---|---|
| 交互导航 Agent | 意图识别、路由分发、上下文管理、结果汇总 |
| 政策计算 Agent | 条件解析、匹配计算、卡点分析、成长规划、联合匹配 |
| 材料工厂 Agent | 清单生成、素材匹配、自动填充、完整性校验、AI/专家预审 |
| 产业协同 Agent | 隐性标签抽取、合作方推荐、协同组网、对接管理 |
| 产业洞察 Agent | 产业图谱、招商建议、趋势预测（园区专用） |

### 6.2 协作流程与通信机制

- 交互导航是统一入口；
- Agent 间通过事件驱动 + 消息队列异步通信；
- 支持链式调用（如：产业协同 → 产业洞察）；
- 结果统一回传交互导航后输出给前端。

消息格式（示例）：


## 7. 数据架构与模型

### 7.1 数据分层

- **基础数据层**：行业/地区/资质字典、标签体系。
- **业务数据层**：用户、企业、人才、园区、政策、素材。
- **应用数据层**：匹配结果、成长路径、申报记录、协同推荐。

### 7.2 核心实体

| 实体 | 说明 | 关键字段 |
|---|---|---|
| User | 账号主体 | ID、手机号、角色、关联实体 |
| Enterprise | 企业主体（含 OPC） | 基础信息、经营、资质、合规、OPC 字段 |
| Talent | 人才主体（含 OPC） | 学历、职称、技能、成果、OPC 字段 |
| Park | 园区主体（含 OPC 社区） | 产业定位、入驻结构、政策发布能力 |
| Policy | 政策主体 | 层级、类型、条件、来源、状态、OPC 标签 |
| Material | 素材主体 | 文件元数据、OCR、结构化提取、有效期 |
| MatchResult | 匹配结果 | 分数、状态、卡点、成长路径、协同结果 |
| Application | 申报记录 | 状态流转、预审结果、反馈凭证 |

## 8. 接口设计

### 8.1 API 规范

- 基础路径：`/api/v1`
- 认证方式：JWT Bearer Token
- 数据格式：JSON（UTF-8）

模块：
- `/auth` `/users` `/enterprises` `/talents` `/policies`
- `/matching` `/materials` `/collaboration` `/insights`
- `/chat` `/parks` `/admin`

### 8.2 核心接口清单（摘要）

#### 8.2.1 材料预审
- `POST /api/v1/materials/{material_package_id}/pre-review`
- `review_type`: `ai | expert`

#### 8.2.2 申报结果反馈
- `POST /api/v1/applications/{application_id}/feedback`
- 字段：`result(passed/rejected/returned)`、`reject_reason`、`grant_amount`、`screenshot(必填)`

#### 8.2.3 AI 对话
- `POST /api/v1/chat/message`（支持 session + 可选流式）
- `GET /api/v1/chat/sessions`（会话列表）

#### 8.2.4 政策匹配
- `POST /api/v1/matching/calculate`
- `GET /api/v1/matching/result/{entity_id}`

#### 8.2.5 企业画像
- `GET /api/v1/enterprises/{enterprise_id}`
- `PUT /api/v1/enterprises/{enterprise_id}`
- `POST /api/v1/enterprises/{enterprise_id}/materials`

#### 8.2.6 园区政策发布
- `POST /api/v1/parks/{park_id}/policies`
- 支持文件上传或手填，入库后返回匹配预览。

## 9. 竞争分析与策略

### 9.1 市场竞争格局（摘要）

- 信息聚合型（政策收录强，但服务浅）
- 政策计算器型（问答/额度计算强，但无执行闭环）
- 综合惠企平台型（广而全，但 AI 垂直深度不足）
- 产业大数据+政策服务型（行业强，但 AI 非专精）
- 政府主导型（权威但区域受限）

### 9.2 本产品差异化

1. **产业深度差异**：AI 合规前置逻辑锁 + OPC 全生命周期。
2. **服务深度差异**：从“匹配”延伸到“成长导航 + 材料工厂 + 预审闭环”。
3. **网络效应差异**：政策服务 + 产业协同双轮驱动，提升高频互动。

### 9.3 竞争策略

- 不做大而全，做 AI 垂直深耕；
- 不做信息平台，做执行服务平台；
- 不做单点工具，做产业连接网络。

## 10. 风险与应对

### 10.1 技术风险（重点）

- LLM 输出稳定性与幻觉风险
- 政策解析准确率风险
- 高并发性能风险
- 数据安全与合规风险
- 向量检索召回效果风险
- 第三方 API 可用性风险
- 自动采集质量风险
- 园区上传内容规范性风险

### 10.2 业务风险（重点）

- 政策变化频繁导致时效风险
- 用户接受度不足
- 园区数据维护意愿不足
- 匹配结果争议
- 申报成功率不及预期
- 权威数据来源受限

### 10.3 项目风险（重点）

- 需求蔓延
- 关键人员流失
- 第三方依赖变动
- 跨团队协作不畅
- 成本超预算
- 上线验收延期

### 10.4 监控机制（关键指标）

| 监控项 | 频率 | 告警阈值 |
|---|---|---|
| LLM 响应准确率 | 实时 | <90% |
| 政策解析准确率 | 每周 | <95% |
| API P95 响应 | 实时 | >3s |
| 政策采集成功率 | 每日 | <90% |
| 申报结果反馈率 | 每周 | <30% |
| OPC 政策覆盖率 | 每月 | <80% |
| 成本偏差 | 每周 | >预算 10% |

## 12. 非功能与合规要求

- 数据安全：敏感字段脱敏、最小权限、审计日志、传输与存储加密。
- 可用性：核心服务高可用、关键链路降级策略、第三方故障熔断。
- 性能：常用接口 P95 < 3 秒（不含大文件上传）。
- 可观测性：全链路日志、指标监控、异常告警闭环。
- 合规声明：本平台仅提供政策辅助服务与材料支持，不替代官方审核结论。

## 13. 里程碑建议（供项目管理）

1. M1（基础能力）：账号体系、画像中心、政策库基础入库与检索。  
2. M2（核心能力）：匹配计算、卡点分析、成长导航、消息通知。  
3. M3（交付闭环）：材料工厂、AI 预审、申报记录。  
4. M4（园区与生态）：园区工作台、智能招商、协同申报。  
5. M5（优化阶段）：策略迭代、效果评估、模型与规则持续优化。

## 14. 验收标准（摘要）

- 政策入库链路可用，支持多源采集与状态管理。
- 画像完善后可产出匹配结果、卡点分析和成长路径。
- 材料工厂支持自动填充、AI 预审、在线修改再预审。
- 申报记录支持状态跟踪与用户结果反馈（含凭证上传）。
- 园区端可发布政策、查看匹配、执行定向推送。
- 核心指标达到上线阈值（准确率、性能、可用性、安全）。
